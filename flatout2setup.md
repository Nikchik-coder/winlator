# 🚀 РУКОВОДСТВО РАЗРАБОТЧИКА RETRONEXUS: Добавление игр

Вот полное, пошаговое руководство (Pipeline) для добавления любой новой игры в маркетплейс RetroNexus. Эту инструкцию можно использовать как официальный стандарт разработчика (SOP). 

> **Цель:** Гарантировать, что любая игра будет работать по принципу «Plug-and-Play» (Скачал ➡️ Нажал ➡️ Играешь), без вылетов, ручной настройки и багов.

---

## 🛑 БАЗОВОЕ ПРАВИЛО RETRONEXUS
**Никаких инсталляторов на устройствах пользователей.** 

Телефоны не должны заниматься распаковкой архивов, установкой библиотек или модов. Вся грязная работа делается заранее на ПК (Ubuntu). В S3 загружается только на 100% готовая, "портативная" папка игры.

---

## 🔍 ФАЗА 1: Поиск и отбор игры

### Ищем правильный формат:
Скачиваем игры с пометками **Portable**, **Unpacked**, **No-Steam** или **GOG**.

### Чего избегать:
*   **ISO-образов** (нужно монтировать).
*   Игр с неотключенным **GFWL** (Games for Windows Live) или **Denuvo**.
*   Старых Steam-кэшей (файлы `.gcf`).

---

## 💻 ФАЗА 2: Подготовка на ПК (Ubuntu + Wine)

Если игра скачана в виде репака (например, `setup.exe`):

1.  **Установка:** Запускаем инсталлятор на Ubuntu через Wine: `wine setup.exe` (для старых игр используем префикс `WINEARCH=win32`).
2.  **Завершение:** Дожидаемся окончания установки на ПК.
3.  **Локация:** Переходим в папку `~/.wine/drive_c/Games/Имя_Игры`.
4.  **Очистка:** Это наша "рабочая директория". Сам инсталлятор (`setup.exe`) можно удалить.

---

## 🛠️ ФАЗА 3: Оптимизация (RetroNexus Tweaks)

Это самый важный этап для стабильной работы на эмуляторе (Winlator Glibc-branch).

### 🖥️ Widescreen Fix (Адаптация под 1280x720)

> [!CAUTION]
> **ЗАПРЕЩЕНО:** Использовать инъекции через `dinput8.dll`, `dsound.dll` или `.asi` скрипты (по умолчанию). Они часто вызывают ошибку `EXCEPTION_ACCESS_VIOLATION` в Box64. Исключения делаются только для конкретных игр с настройкой `overrides` в конфиге Winlator.

*   ✅ **МЕТОД 1 (UniWS):** Пропатчить сам исполняемый файл `.exe` через утилиту UniWS на ПК (самый безопасный метод).
*   ✅ **МЕТОД 2 (Аргументы):** Использовать параметры запуска (например, `-width 1280 -height 720` для движка Source/Half-Life).
*   ✅ **МЕТОД 3 (Реестр):** Создать файл `widescreen.reg` в папке игры (как для Warcraft 3).

### 🧹 Очистка от мусора
*   Удаляем файлы деинсталляторов (`unins000.exe`, `unins000.dat`).
*   Удаляем папки `DirectX`, `VCRedist`, `Soundtrack`, неиспользуемые языки озвучки.

### 💿 No-CD
Убеждаемся, что оригинальный `.exe` заменен на взломанный (кряк), чтобы игра не просила диск.

---

## 📦 ФАЗА 4: Создание архива для S3

1.  Открываем папку с готовой игрой.
2.  Выделяем **все файлы внутри папки** (не саму папку, а её содержимое).
3.  Создаем стандартный **ZIP-архив** (например, `game_retronexus.zip`).
4.  Загружаем этот `.zip` и картинку (логотип) в бакет Cloudflare R2 / AWS S3. 
5.  Получаем прямые ссылки (URL).

## 💥 FlatOut 2

### Статус: НЕ РЕШЕНО (диагностика)

FlatOut 2 вылетает с `EXCEPTION_ACCESS_VIOLATION` (code=c0000005) в Box64 JIT-коде. Адрес краша стабильно `7Axx26A9` — смещение `26A9` фиксировано, база `7Axx` меняется от запуска к запуску (ASLR). Это указывает на конкретную инструкцию внутри JIT-скомпилированного кода, а не на случайную memory corruption.

### Хронология краша (из logcat)

```
RPC_S_SERVER_UNAVAILABLE exception (code=6ba)        ← RpcSs сервис (решено)
init_peb starting FlatOut2.exe in experimental wow64 mode
DXVK: No state cache file found
D3D9DeviceEx::SetRenderState: Unhandled render state 26
Compiling shader FS_13f371a6...                      ← Последнее действие перед крашем
EXCEPTION_ACCESS_VIOLATION (code=c0000005) on thread 017c
wine: Unhandled page fault on read access to 7A9126A9 at address 7A9126A9
[BOX64] winedbg detected, not launching it!
```

### Корневые проблемы

**Проблема 1 (решена): RPC_S_SERVER_UNAVAILABLE**
Wine-сервис `RpcSs` не запускался вовремя. FlatOut 2 под win7 использует COM/DCOM для аудио (через `mmdevapi` → `IAudioSessionControl`). Решено переходом на `winxp` + ранней записью в реестр.

**Проблема 2 (решена): IndirectSound несовместим с winxp**
Нативный `dsound.dll` (IndirectSound) требует XAudio2/mmdevapi для перечисления аудиоустройств. Под winxp не работает. Решено удалением IndirectSound и переходом на Wine builtin dsound (`dsound=b`).

**Проблема 3 (НЕ РЕШЕНА): Box64 dynarec crash в wow64**
Краш происходит в JIT-скомпилированном коде (адрес `7Axx26A9`) сразу после компиляции шейдеров DXVK. Крашится не главный поток, а вторичный (номер потока меняется каждый запуск: `0194`, `017c`, `0184` и т.д.). Ни STABILITY preset Box64, ни отключение esync/LAA, ни отключение DXVK state cache/async не помогают.

**Текущая гипотеза:** Баг в Box64 dynarec при трансляции wow64-thunks для 32-bit .exe. Сейчас тестируется `BOX64_DYNAREC=0` (чистый интерпретатор) для подтверждения.

### Что не работает на устройстве

- **WineD3D**: Нет `libEGL.so.1` в rootfs → WineD3D не может создать GL-контекст. На данном устройстве только DXVK (Vulkan) работает.
- **mmdevapi= (disabled)**: Вызывает null pointer crash — игра ожидает валидный COM-объект от `CoCreateInstance`.
- **dsound=n,b с IndirectSound**: IndirectSound не может перечислить аудиоустройства через XAudio2/mmdevapi под winxp.

Технический анализ и стратегия отладки дефектов запуска FlatOut 2 в среде эмуляции Winlator: Архитектура Box64, механизмы WOW64 и проблемы синхронизации в JIT-компилятореПроблема запуска наследуемого программного обеспечения x86-архитектуры на современных мобильных платформах ARM64 представляет собой одну из наиболее сложных задач в области системного программирования и виртуализации. В контексте эмулятора Winlator, объединяющего слои Wine, Box64 и графические трансляторы вроде DXVK, запуск игры FlatOut 2 (2006) сталкивается с критическими ошибками доступа к памяти, которые требуют глубокого понимания механизмов динамической рекомпиляции и работы системных вызовов в гибридных средах. Данный отчет представляет собой исчерпывающий анализ текущего состояния отладки, классификацию выявленных дефектов и предложение комплексной методологии их устранения.Теоретические основы многослойной эмуляции в WinlatorДля интерпретации причин возникновения ошибки EXCEPTION_ACCESS_VIOLATION (c0000005) необходимо декомпозировать архитектуру стека, в котором функционирует FlatOut 2. Эмуляция на устройствах Android через Winlator строится на взаимодействии нескольких изолированных, но взаимозависимых подсистем. На нижнем уровне находится ядро Android (Linux), обеспечивающее управление ресурсами процессоров Snapdragon (архитектура ARM64/AArch64). Поверх него развертывается среда Glibc, которая служит фундаментом для работы Wine. Ключевым компонентом является Box64 — динамический рекомпилятор (dynarec), преобразующий инструкции x86_64 и x86 в нативный код ARM64 в режиме реального времени.Механизм динамической рекомпиляции и его ограниченияBox64 не просто интерпретирует код, а анализирует блоки инструкций x86, оптимизирует их и кэширует в виде нативных фрагментов кода ARM64. Этот процесс критически зависит от предсказуемости доступа к памяти. Ошибки типа Unhandled page fault часто возникают, когда рекомпилированный блок пытается обратиться к адресу, который был изменен другим потоком или защищен механизмом ASLR (Address Space Layout Randomization). В случае с FlatOut 2, стабильный адрес краша 7Axx26A9 с фиксированным смещением 26A9 указывает на то, что проблема локализована в конкретном сегменте кода, вероятно, связанном с переходом между 32-битным адресным пространством приложения и 64-битными библиотеками эмулятора (так называемые thunks).Специфика WOW64 в мобильных средахРежим WOW64 (Windows-on-Windows 64) в Wine позволяет запускать 32-битные приложения в 64-битной среде. Однако на мобильных устройствах Android существуют специфические ограничения адресного пространства (обычно 39 или 48 бит), что может конфликтовать с ожиданиями старых Windows-приложений, которые рассчитывают на плоскую модель памяти объемом до 4 ГБ. Несмотря на то что большинство программ справляется с этим, игры с интенсивным использованием динамических библиотек и сложными структурами данных (как движок Bugbear во FlatOut 2) могут провоцировать некорректное поведение JIT-компилятора при маппинге памяти.Диагностика критической ошибки доступа к памятиАнализ логов (logcat) показывает, что FlatOut 2 инициирует падение сразу после завершения компиляции шейдера FS_13f371a6. Это позволяет выдвинуть гипотезу о наличии состояния гонки (race condition) между потоком рендеринга и основным потоком обработки логики. В момент, когда графический драйвер Turnip через DXVK завершает подготовку шейдера и возвращает управление игре, Box64 пытается выполнить переход в соответствующий блок кода, где и происходит нарушение прав доступа.Феномен адреса 7Axx26A9Исследования сообщества и анализ репозитория Box64 указывают на то, что диапазон адресов 0x7a000000 - 0x7c000000 часто используется для специфических операций эмуляции или маппинга системных библиотек. Примечательно, что в файле конфигурации box64rc для некоторых приложений принудительно отключается динамическая рекомпиляция (BOX64_NODYNAREC) именно в этом диапазоне.Параметр памятиЗначение во FlatOut 2Техническое влияниеАдрес краша (Base)7Axx (динамический)Указывает на область памяти, управляемую эмуляторомСмещение (Offset)26A9 (статическое)Соответствует конкретной инструкции в JIT-блокеКод исключенияc0000005Нарушение сегментации или попытка записи в Read-Only памятьСтатус ASLRВключенВызывает изменение базового адреса при каждом запуске Данная таблица подтверждает, что ошибка носит систематический характер и не является результатом случайного повреждения памяти. Скорее, это баг трансляции конкретной инструкции или последовательности операций в WOW64-слое Box64.Влияние логирования на стабильность (Heisenbug)Существует задокументированный эффект, при котором установка переменной окружения BOX64_DYNAREC_LOG=2 позволяет избежать крашей и зависаний. Это классический пример "Гейзенбага", когда внесение задержек, связанных с выводом логов в консоль, изменяет тайминги выполнения потоков. В многопоточных приложениях, таких как FlatOut 2, это может предотвращать одновременный доступ к критическим секциям кода, которые еще не были полностью обработаны JIT-компилятором.Анализ аудиоподсистемы и сервисов RPCОдной из успешно решенных проблем в процессе отладки стал запуск сервиса RpcSs. В Windows-среде этот сервис отвечает за удаленный вызов процедур и является критически важным для работы COM-объектов.Переход на WinXP и mmdevapiИзначально FlatOut 2 запускался с профилем Windows 7, что заставляло игру использовать интерфейс mmdevapi для работы со звуком. Этот программный интерфейс значительно сложнее в эмуляции, чем классический DirectSound. Хотя асинхронный режим (DXVK_ASYNC=1) обычно повышает плавность игры, в условиях эмуляции он может приводить к тому, что игра пытается отрисовать кадр, используя шейдер, который еще не загружен в память GPU. Отключение кэша состояний (DXVK_STATE_CACHE=disable) и ограничение числа потоков компиляции (DXVK_NUM_COMPILER_THREADS=1) являются важными диагностическими шагами для исключения графического драйвера из списка причин сбоя.Математическая модель производительности эмуляцииЭффективность работы игры в эмуляторе можно представить в виде функции от накладных расходов на каждом уровне. Общая производительность $P$ определяется как:$$P = \frac{P_{native}}{1 + C_{dynarec} + C_{api} + C_{driver}}$$Где:$P_{native}$ — теоретическая производительность на нативном железе.$C_{dynarec}$ — стоимость трансляции инструкций x86 -> ARM (для Box64 составляет от 15% до 40% ).$C_{api}$ — накладные расходы Wine на трансляцию системных вызовов.$C_{driver}$ — стоимость трансляции D3D9 -> Vulkan.Во FlatOut 2 основная нагрузка ложится на $C_{dynarec}$ из-за интенсивных физических расчетов. Любые ошибки в блоке dynarec приводят не к снижению $P$, а к полному отказу системы (crash).Тонкая настройка Box64: Strong Memory Model и Safe FlagsДля решения проблемы краша в WOW64-слое необходимо обратиться к расширенным настройкам Box64. Многие старые игры для Windows полагаются на "сильную" модель памяти x86, которая гарантирует порядок выполнения операций чтения и записи. Процессоры ARM используют "слабую" модель, что может приводить к тому, что данные в памяти обновляются не в том порядке, который ожидает игра.Значение Strong Memory Model (STRONGMEM)Использование BOX64_DYNAREC_STRONGMEM позволяет эмулировать строгое поведение x86 на ARM-процессорах. Для FlatOut 2 рекомендуется протестировать значения 1 или 2. Уровень 3 обеспечивает максимальную совместимость, но значительно снижает FPS, что может быть неприемлемо для гоночного симулятора.Оптимизация обработки исключений и сигналовНа устройствах Android работа с сигналами сегментации (SIGSEGV) ограничена. Box64 имеет встроенные механизмы перехвата таких сигналов для предотвращения падения всего приложения. Этот метод позволяет сохранить высокую производительность для основного кода игры, переключаясь на медленную, но стабильную интерпретацию только в критических системных областях.Шаг 3: Исправление аудио-стека через Native DLLКак было выявлено в ходе исследования, mmdevapi=b недостаточно для полной стабильности. Необходимо интегрировать в архив игры нативные библиотеки xapofx1_5.dll и x3daudio1_7.dll из состава DirectX Redistributable и прописать их в WINEDLLOVERRIDES. Это устранит потенциальные сбои в COM-интерфейсах, которые могут косвенно влиять на стабильность потоков.Заключение и рекомендацииПроблема запуска FlatOut 2 на Winlator является многофакторной. Если этот метод окажется успешным, он станет стандартом для добавления FlatOut 2 в базу данных Supabase с соответствующим config_preset, обеспечивающим работоспособность игры по принципу Plug-and-Play.
